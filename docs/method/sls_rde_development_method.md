# SLS + RDE に基づく開発手法（Koguchi）

本書は、**Koguchi** の設計・実装・ドキュメントを進めるとき、**Semantic Lineage System（SLS）** の考え方と **Resonant Deviation Evaluator（RDE）** 型のレビューをどう運用するかを説明する実践手引きである。**規範仕様ではない**。不変条件・スキーマ・状態機械の正本は [`docs/adr/`](../adr/) および設計仕様書に置く。

元文書: [kotonoha-docs — SLS + RDE に基づく開発手法](https://github.com/zyx-corporation/kotonoha-docs/blob/main/ja/method/sls_rde_development_method.md)

---

## 1. 目的

**観念の実践（dogfooding）:** Koguchi が目指す「すべての副作用を通過させる単一の隘路」「来歴の監査可能性」を、将来の成果物だけでなく **今日の開発プロセス**でも体現する。

- 変更を **diff だけ**として終わらせない。
- **何が保存・変換・補完され、何が未解決のまま残り、何が失われ、どこに逸脱リスクがあるか** を、PR・レビュー・記録で落とす。
- その結果を **次の設計・実装・ADR 更新判断**へ返す（次回更新方針）。

対象は **設計仕様書**、**スキーマ・不変条件**、**実装コード（`src/koguchi/`）**、**テスト**、および **ADR** 全体である。

---

## 2. 用語（本書レベルの最小定義）

| 用語 | 本書での意味 |
| --- | --- |
| **SLS** | **意味 lineage** を制度として扱う層。Git・Issue・ADR を否定せず、その上に「意味・喪失・逸脱・責任」を載せる発想。 |
| **RDE レビュー** | 変更を **観測カテゴリ**（§3）に沿って読むレビュー。**テスト通過だけ・型チェックだけ**で終わらせない。 |
| **ΔM（意味変化）** | 文字の増減だけでなく、不変条件・責任境界・設計意図の変化として捉える差分。Koguchi では特に INV-1b（二相記録）と INV-1c（迂回検出）への影響を注視する。 |
| **隘路（Chokepoint）** | Koguchi の中核概念。副作用が必ず通る単一の通過点。コードの変更がこの単一性を崩す方向に働くとき、それは ΔM が大きい変更である。 |

RDE を **採点器・万能フィルタ・人間の責任の代替**とみなさないこと。Koguchi は `UNCONFIRMED` という正直さを設計に刻んでいる——開発手法も同様に、未確定を未確定として残す。

---

## 3. RDE の観測カテゴリ（日常運用への写像）

PR・設計変更・ADR 更新をレビューするとき、少なくとも次を **後から読めるか** を確認する（順序固定ではない）。

1. **保存された要素** — 意図的に維持した不変条件・約束（例: INV-1b の二相記録、atomic write の「全か無か」保証）。
2. **変換された要素** — 言い換え・構造変更で意味を載せ替えた箇所（暗黙のすり替えでないか。例: `FAILURE` の意味を「副作用なし」から「成功完了を確認できなかった」へ再定義した v1.1 の変更）。
3. **補完された要素** — 新しく付与された説明・前提・構造（例: `side_effect_observed` フィールドの追加）。
4. **未解決のまま残した要素** — 決め切らず残した論点・緊張（例: `partial` / `unknown` の将来ツール用余地、`expected_result_digest` の非決定論ツールへの一般化）。
5. **失われた要素／喪失** — 整理・実装の便宜で薄まった射程・痛み・責任・問い（削除行だけで代替しない。例: reconciliation が「確定ではなく最尤推定」であるという限界の明示）。
6. **逸脱リスク** — 隘路の単一性の崩壊、「安全な Tool Runtime」への縮退（§9 縮退防止フック参照）、不変条件の形骸化。
7. **次回更新方針** — 次の実装フェーズ・ADR 更新に何を渡すか（一回きりの TODO で終わらせない）。

実務上、**すべてに長文で埋める義務**はない。**重大な喪失や逸脱が「見えない」状態**だけは避ける。

## 4. Git・Issue・ADR との役割分担

| 仕組み | 得意なこと | SLS + RDE がメソッドとして足すこと |
| --- | --- | --- |
| **Git** | テキスト差分・コミット履歴 | 「何が変わったか」に加え **意味として何が起きたか**（ΔM）を PR 本文で補う。 |
| **Issue** | 未解決作業・議論 | クローズ時に **未解決・喪失** が議論から消えていないか確認する。 |
| **ADR** | アーキテクチャ判断の記録 | 判断の背景・トレードオフ・意図的に延ばした論点を残す。変更時は新 ADR で supersede する（上書きしない）。 |
| **テスト（§10 赤テスト）** | 不変条件の緑・赤 | テストは「機能が動く」ではなく「不変条件が守られているか」を突く。赤テストが緑になることが ΔM の実装レベルでの決着。 |

**Git が「変化」を残すなら、RDE 志向のレビューは「変化が不変条件と責任境界に何をしたか」を問う。**

## 5. リポジトリ内の流れ

Koguchi は現時点で単一リポジトリである。役割の分離は**ディレクトリ境界**で行う。

| ディレクトリ | この手法での役割 |
| --- | --- |
| `docs/adr/` | **判断の正本**。不変条件・スキーマ・設計原則の変更は ADR として記録する。 |
| `docs/method/` | **手法の説明**（本書）。規範仕様ではない。ADR で参照される。 |
| `src/koguchi/` | **実装**。スキーマ・不変条件の変更は ADR へのトレースを意識する。 |
| `tests/` | **不変条件の物理的証拠**。テストは §10 の赤テストを起点とし、不変条件を突く構造を維持する。 |

**`docs/method/` で規範語義を静かに書き換えない。** 規範的な変更は ADR で行う。

## 6. 習慣（軽量）

**作成者**

- 大きめの PR では **意図** と **スコープ外** を書く。
- 不変条件（INV-1/1a/1b/1c）への影響と **意図的に延ばした論点** を diff に埋もれさせない。
- `UNCONFIRMED` という戻り値が示す誠実さ——「わからないことをわからないと返す」——を、PR 本文でも実践する。

**レビュア**

- 設計・スキーマ・不変条件の変更では、**七つの観測カテゴリ**（§3）をプロンプトとして使う。
- **隘路の単一性**を崩す変更（Tool Proxy を迂回する経路の追加など）には明示的に抵抗する。
- 「縮退防止フック（`intent` / `decision_ref` / `context_ref`）が空のまま閉じようとする変更」は逸脱リスクとして flag する。

**メンテナ**

- Phase が進むにつれて「安全な Tool Runtime」への縮退リスクが高まる。逸脱リスクを repeating agenda に含める。
- 空フックが実装で埋まるタイミングを見計らい、ADR を更新する。

---

## 7. 本手法でないもの

- **ADR の代替ではない**（不変条件・判断の正本は ADR）。
- **テストが RDE 全体を自動実行済み**であることを前提にしない（テストは不変条件の証拠であり、設計意図の記録ではない）。
- **reconciliation が完全な監査**を意味しない（§13 既知の限界のとおり、診断は最尤推定）。
