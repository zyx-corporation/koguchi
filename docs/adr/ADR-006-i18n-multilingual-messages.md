## 背景

Koguchi は AI エージェントの副作用を監査可能な来歴へ変換するライブラリであり、その出力（エラーメッセージ、診断詳細、監査レポート）はエージェント運用者・開発者・監査者という複数層の人間に読まれる。現状、全ユーザー向け文字列が日本語でハードコードされており、日本語圏外の運用者が Koguchi の出力を理解できない。

本 ADR は、Koguchi の i18n（国際化）方針と、それを実現するコーディング規約を定める。

---

## 決定 A: 三層の言語ポリシー

**Koguchi のテキストは三層に分類し、それぞれ異なる言語ポリシーを適用する。**

| 層 | 内容 | 言語 | i18n 対象 |
| --- | --- | --- | --- |
| **コード識別子** | クラス名、関数名、変数名、フィールド名、メッセージキー | **英語** | 不要（普遍） |
| **開発者向け文書** | docstring、コメント、ADR | **日本語**（ソース言語） | 不要（開発者は日本語） |
| **ユーザー向け文字列** | エラーメッセージ、診断詳細、監査出力 | **多言語**（ja/en/zh-CN/zh-TW/ko） | **必須** |

### 理由

- **コード識別子を英語にする**のは Python エコシステムの普遍的な慣習であり、すでに Koguchi はこれを満たしている。
- **開発者向け文書を日本語にする**のは現在の開発チームの使用言語であり、ソースコードの可読性を維持する。翻訳コストをかけてまで英語化する利益は現時点で薄い。
- **ユーザー向け文字列を多言語化する**のは、Koguchi が AI エージェント運用の監査チェーンに組み込まれる際、日本語圏外の運用者がエラーメッセージや診断結果を読める必要があるため。

## 決定 B: シンプルな JSON ベースのメッセージカタログ

**Python 標準の `gettext` は使わず、独自の軽量 JSON メッセージカタログを採用する。**

### メッセージカタログ構造

```
locales/
├── ja.json        # ベース言語（フォールバックの最終段）
├── en.json
├── zh-CN.json     # 簡体字中国語
├── zh-TW.json     # 繁体字中国語（台湾）
└── ko.json
```

各ファイルはメッセージキーをキーとするフラットな辞書：

```json
{
  "err.envelope_required": "ActionEnvelope は必須です",
  "err.workspace_boundary": "{{target}} は workspace_dir 外です",
  "reconcile.target_not_found": "{{target}} は存在しない",
  "reconcile.target_exists_unconfirmed": "{{target}} が存在する（digest={{digest}}）"
}
```

### メッセージキー命名規則

- プレフィックスで層を区別: `err.`（エラー）、`reconcile.`（診断）、`chain.`（検証）
- スネークケース（`snake_case`）、英語
- プレースホルダは `{{param}}` 形式（Python `str.format()` 互換）

### フォールバックチェーン

```
要求ロケール → ja → キー名そのまま
```

1. 要求ロケールの JSON を探す
2. なければ `ja`（ベース言語）にフォールバック
3. キーが JSON に存在しなければ、キー名自体を表示（開発者向けシグナル）

### ロケール指定方法

優先順位:
1. 環境変数 `KOGUCHI_LOCALE`
2. プログラムからの `set_locale(locale)` 呼び出し
3. デフォルト `ja`

### 理由

- **gettext 非採用**: `.po`/`.mo` のコンパイルステップ・バイナリ配布・`xgettext` 抽出ワークフローは、Koguchi の現在の規模（~300行）に対して過剰。JSON は Python が標準で読め、翻訳者の学習コストが低い。
- **フラット構造**: 深いネストよりフラットなキーの方が翻訳ファイルの diff が読みやすく、欠落検出も容易。
- **ベース言語 = ja**: 開発チームの母語であり、最も正確な表現が保証される。日本語が読めるなら常に最も信頼できるメッセージが得られる。

## 決定 C: API — `t(key, **params)` 関数

**全ユーザー向け文字列の生成は `koguchi.i18n.t(key, **params)` を通す。**

```python
from koguchi.i18n import t

raise EnvelopeRequiredError(t("err.envelope_required"))

detail = t("reconcile.target_not_found", target=str(target))
```

### コーディング規約

1. **ユーザー向け文字列にハードコードした自然言語文字列を置かない。** 必ず `t(key, **params)` を使う。
2. **新しいユーザー向け文字列を追加する場合、`ja.json` に必ずエントリを追加する。** 他言語は必須ではない（欠落時は ja にフォールバック）。
3. **docstring・コメントは引き続き日本語で書く。** これらは i18n 対象外。
4. **メッセージキーは英語のスネークケース。** 翻訳者がキーの意味を類推できるようにする。
5. **プレースホルダは `{{param}}` 形式。** `str.format()` と競合しないよう中括弧を二重にする。

## 意図的に延ばした論点

| 論点 | 延ばした理由 |
| --- | --- |
| 翻訳ファイルの動的リロード | 現在は起動時読み込み。長期運用で翻訳更新が必要になった時点で対応。 |
| 複数形対応（ngettext） | Koguchi の現在のメッセージに可算名詞の複数形が存在しない。必要になった時点で `t_plural()` を追加。 |
| ロケールごとの日付/時刻フォーマット | `datetime.isoformat()` で統一済み。ロケール固有フォーマットは表示層の責務。 |
| L10n（地域化） | i18n 基盤が整った後、実際の運用フィードバックに基づいて翻訳品質を改善する。 |

---

## 次回更新方針

- 翻訳ファイルが 50 エントリを超えた時点で、翻訳管理の仕組み（CI での欠落チェック等）を検討。
- 他言語翻訳の品質レビューは、各言語のネイティブ話者がいる時点で実施。
