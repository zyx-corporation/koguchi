# v0.10 Backend-specific Reconciliation Design Gate

## Status

**Design Gate** — not an implementation.

This document defines the scope and boundaries for backend-specific reconciliation in v0.10. No reconciliation code is implemented here.

## Purpose

Backend-specific reconciliation is a bounded observation and comparison process for backend outputs, audit records, and expected execution traces. It is not a control plane, not a security sandbox, and not an autonomous recovery mechanism.

backend-specific reconciliation は、backend の出力・監査記録・期待された実行痕跡を比較するための限定された観測・照合プロセスである。制御平面ではなく、セキュリティサンドボックスでもなく、自律復旧機構でもない。

## Scope

This design gate covers:

- Backend-specific reconciliation responsibility boundaries
- Separation of Scheduler and Backend Reconciliation
- Relationship between ToolProxy, ExecutionBackend, AuditStore, and ResultStore
- Minimal specification for filesystem diff reconciliation
- Reconciliation result classification
- Boundary between read-only observation and write action
- Handling of failure, mismatch, and stale results
- Mapping to audit events

## Non-goals

- No backend-specific reconciliation implementation
- No automatic recovery
- No automatic retry policy
- No production control plane
- No security sandbox
- No write-side enforcement
- No remote API
- No web server
- No crypto sealing
- No Rust protocol stabilization
- No backend mutation
- No autonomous scheduler decision

## Key Distinctions

### Scheduler

Decides when a reconciliation check should be requested. The Scheduler does not judge backend correctness.

### Backend-specific Reconciliation

Compares observed backend state, expected outputs, and audit records. This is observation and comparison, not repair or control.

### ExecutionBackend

Executes backend-specific operations through controlled paths (ToolProxy → PolicyGate → ServiceRuntime → RuntimeBoundary → ExecutionBackend).

### ToolProxy

Remains the single side-effect chokepoint. All managed side effects pass through ToolProxy.

### AuditStore

Records what was requested, observed, and compared (JSONL append-only).

### ResultStore

Stores reconciliation outputs (JSONL append-only, separate from audit log).

## Minimal Reconciliation Model

### ReconciliationInput

```
- backend_id
- execution_id
- expected_artifact_ref
- observed_artifact_ref
- audit_event_refs
- result_store_refs
- policy_context_ref (optional)
```

### ReconciliationOutput

```
- status
- summary
- observed_differences
- missing_artifacts
- stale_artifacts
- audit_gaps
- recommended_review_focus
```

### Status values

| Status | Meaning |
|--------|---------|
| `matched` | Expected and observed state align |
| `mismatch_observed` | Differences found — review focus, not action |
| `missing_expected_artifact` | Expected artifact not found in observed state |
| `missing_observed_artifact` | Observed artifact has no corresponding expectation |
| `stale_observation` | Observation is older than acceptable threshold |
| `audit_gap` | Audit record missing for expected execution |
| `inconclusive` | Cannot determine alignment |

Status values are review signals, not control actions. `mismatch_observed` means "review focus here" — not "repair this."

## Filesystem Diff Reconciliation

The first concrete reconciliation target is filesystem diff.

### Scope

- Expected file list vs observed file list
- Expected hash vs observed hash (optional)
- Generated artifact paths
- Ignored paths
- Timestamp tolerance

### Diff categories

| Category | Meaning |
|----------|---------|
| `added` | File exists in observed but not in expected |
| `removed` | File exists in expected but not in observed |
| `modified` | File exists in both but content/hash differs |
| `unchanged` | File matches expected |
| `ignored` | Path in ignore list |
| `unknown` | Cannot determine |

### Forbidden actions

- Automatic deletion
- Automatic overwrite
- Automatic rollback
- Automatic commit
- Automatic deploy

## Boundary Language

Consistent with v0.9 terminology:

- ToolProxy, PolicyGate, RuntimeBoundary, ServiceRuntime, ExecutionBackend
- AuditStore, ResultStore
- read-only, not a security sandbox, not a control plane
- observation plane, review focus

## RDE Consistency Check

### Preserved

- ToolProxy remains the side-effect chokepoint.
- Koguchi remains not a security sandbox.
- Koguchi remains not a production control plane.
- Dashboard remains an observation plane.

### Transformed

- v0.9 known limitation "backend-specific reconciliation is unimplemented" becomes a v0.10 design target.

### Complemented

- Scheduler / Backend Reconciliation / ExecutionBackend / AuditStore / ResultStore boundaries clarified.
- Minimal reconciliation input/output model defined.
- Filesystem diff reconciliation scope defined.

### Intentionally unresolved

- Actual reconciliation implementation.
- Rust protocol stabilization.
- Remote API / web server.
- Crypto sealing.
- Automatic recovery policy.

### Deviation risks

- Reconciliation may be mistaken for control-plane behavior.
- Mismatch status may be mistaken for enforcement.
- Filesystem diff may be mistaken for automatic repair.

### Next update policy

- Keep v0.10 design gate non-implementation.
- Add examples only after boundary language is stable.
