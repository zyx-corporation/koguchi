# v0.18 Schema Implementation Design Follow-up

## Status

**Design Follow-up** — no schema files or validation implemented.

v0.18 narrows the first stable schema implementation scope for Koguchi reconciliation records. It does not add JSON Schema files or protocol implementation. It defines which candidate surfaces should be implemented first and which surfaces remain intentionally unstable.

## First Implementation Scope

**Target:** `DiffKind`, `ReconciliationStatus`, `FilesystemDifference`, `ReconciliationResultRecord`

Rationale: ResultRecord is the center of CLI inspection, persistence, and review focus. Stabilize result schema before AuditRecord, ToolProxy, or Scheduler surfaces.

## Deferred Surfaces

`ReconciliationAuditRecord`, `ToolProxyReconciliationRequest/Response`, `SchedulerReconciliationRequest/Result`, `CliInspectionResult` — deferred because integration policies are not yet finalized.

## Required Fields: ReconciliationResultRecord

```
schema_version: "koguchi.reconciliation.result.v0"
result_id, request_id, status, summary
observed_differences, missing_artifacts, stale_artifacts, audit_gaps
recommended_review_focus
boundary: {review_focus_only: true, correctness_decision: false,
           enforcement: false, approval: false, rejection: false}
```

## Required Fields: FilesystemDifference

```
path, kind, expected_sha256 | null, observed_sha256 | null
```

## Enum Policies

**DiffKind:** added, removed, modified, unchanged, ignored, unknown. Invariant: removed ≠ delete, modified ≠ overwrite, added ≠ adopt.

**ReconciliationStatus:** matched, mismatch_observed, missing_expected_artifact, missing_observed_artifact, stale_observation, audit_gap, inconclusive. Invariant: matched ≠ approval, mismatch_observed ≠ failure, audit_gap ≠ security verdict, inconclusive ≠ correctness decision.

## Unknown Field / Enum Policy

- Unknown optional fields: consumers may ignore.
- Unknown enum values: must not be silently reinterpreted. Fail validation or map to explicit unknown state only if schema version allows.

## Boundary Field Policy

Boundary fields are semantic guardrails, not decoration. Removing them is breaking. Flipping polarity is a critical distortion. Making them optional requires separate design review.

## JSON Schema Spike Requirements

Before implementation: add schema for ResultRecord only, add validation tests (valid, missing required, invalid enum, matched≠approval, mismatch≠failure). Do not add ToolProxy/Scheduler/CLI/production persistence/wire protocol in same PR.

## Non-goals

No JSON Schema, validation, protocol implementation, production persistence, CLI stabilization, ToolProxy/Scheduler/CLI schema stabilization, Rust protocol, remote API, crypto sealing.

## RDE

Preserved: matched≠approval, mismatch≠failure, removed≠delete, observation artifacts, boundary guardrails, not control plane/sandbox.
Transformed: v0.17 broad scope → narrow first implementation.
Complemented: First target, deferred surfaces, required fields, enum/unknown/boundary policies.
Next: ResultRecord JSON Schema spike (only ResultRecord).
